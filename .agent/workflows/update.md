이 워크플로우는 구현된 기능이 프로젝트의 어느 부분에 속하는지 판단하고, 기존 문서 업데이트 또는 신규 기능 명세서 생성을 통해 프로젝트의 '공유 메모리'를 최신 상태로 유지하는 과정을 안내합니다. 시니어 엔지니어의 관점에서 시스템의 확장성과 유지보수성을 고려하여 문서를 관리합니다.

### 📝 업데이트 세부 지침 (Rules)
- **문서 탐색 및 판단 (Find vs Create)**: 구현된 기능의 성격(기획/데이터/로직/도구)에 따라 `document-map.md`를 참고하여 최적의 부모 문서를 선택합니다. 기존 문서 범주에 속하지 않는 대규모 신규 시스템인 경우 신규 MD 파일을 생성합니다.
- **관계 정의 중심**: 단순히 기능을 나열하는 대신, 해당 기능이 참조하는 매니저와 사용하는 데이터(SO)를 명확히 기술합니다.
- **Mermaid 다이어그램 필수 업데이트**: 시스템 상호작용이 발생하는 경우 해당 문서 혹은 `architecture.md`의 [3. 상호작용] 섹션 다이어그램을 반드시 업데이트합니다.
- **데이터 흐름 추적**: 새로운 데이터 구조(SO 등)가 도입되었다면 [Architecture/data-structures.md]에 명시합니다.
- **자동화 인벤토리 최신화**: `Execution/` 폴더에 새로운 툴이나 템플릿이 추가되었다면, 반드시 [Execution/README.md]의 도구 목록과 메뉴 경로를 업데이트합니다.
- **문서 지도 최신화 (Essential)**: 새로운 문서가 생성되었거나 문서 역할이 변경되었다면, `Directives/Architecture/document-map.md`를 최우선으로 업데이트합니다.
- **관련 지침 동기화**: 프로젝트 운영 방식이나 AI의 페르소나/작업 규칙이 변했다면 `Directives/Agent/agent-rules.md`를 함께 검토하고 업데이트합니다.
- **Self-Cleaning**: '5. 앞으로 완성해야 할 부분'에 있던 항목이 구현되었다면, 해당 항목을 삭제하고 본문에 내용을 반영합니다.

### 1단계: 추가된 기능 확인
// turbo
1. 최근 변경된 파일이나 구현된 새로운 기능을 확인합니다. 만약 특정 기능에 대해 명시하고 싶다면 지금 말씀해 주세요.

### 2단계: 관련 문서 및 대상 분석
2. `Directives/Architecture/document-map.md`를 읽어 프로젝트의 문서 체계와 각 문서의 역할, 업데이트 우선순위를 파악합니다.
3. 구현된 기능에 가장 적합한 **기존 문서**를 선정하거나, 필요시 **새로운 기능 명세용 문서**를 작성할지 결정합니다.
   - 예: 새로운 무기는 `content_spec.md`, 새로운 매니저는 `architecture.md`, 대규모 신규 시스템은 `feature-xxx.md` 생성.

### 3단계: 업데이트 내용 제안 및 정리
4. 위 **[업데이트 세부 지침]**과 각 문서의 **[역할]**에 맞춰 문서의 적절한 섹션에 배치하여 업데이트안을 작성합니다.
   - **문서 매핑 유지**: `architecture.md`의 코드↔기획 매핑 테이블이 변경되었다면 이를 반드시 반영합니다.
   - **기능 설명 추가**: 역할과 의존성(Manager, SO) 혹은 도구의 권한을 명시합니다.
   - **시각화 유지**: Mermaid 코드를 수정하여 전체 시스템 구조와 흐름의 무결성을 유지합니다.
   - **추가 개선사항 제안**: 단순히 기능을 업데이트하는 것을 넘어, 시스템적으로 더 나아질 수 있는 부분(리팩토링, 보안, 성능 등)이 있다면 반드시 사용자에게 보고하고 제안합니다.

### 4단계: 사용자 승인 및 적용
5. 제안된 변경 사항을 사용자에게 보여주고 승인을 받습니다. 승인 후 파일에 최종 반영합니다.
